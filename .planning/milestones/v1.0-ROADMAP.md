# Milestone v1.0: MGC Research Workflow

**Status:** SHIPPED 2026-04-09

## Overview

v1.0 delivered a complete local MGC futures research workflow on Windows: Databento ingestion into a Nautilus catalog, event-driven backtesting, rule-based strategy execution, Optuna optimization with holdout evaluation, and final workflow hardening for repeatable operator use.

## Milestone Summary

**Key Accomplishments**

- Delivered validated Databento ingestion with counts, date-range reporting, and warning classification around the real MGC catalog load.
- Added a reusable Nautilus backtest runner with result extraction, trade logs, and persisted artifact bundles.
- Implemented the production MGC rule-based strategy and a dedicated risk-management layer aligned with Nautilus native infrastructure.
- Added Optuna optimization with ranked in-sample results, best-run exports, and separate holdout evaluation artifacts.
- Hardened the operator workflow with shared preflight validation, a `health` command, safer artifact persistence, and a practical usage guide.

**Key Decisions**

- Use a structured CLI-driven Python project instead of notebooks or ad hoc scripts.
- Keep Nautilus Trader native infrastructure as the foundation rather than building a custom simulator.
- Preserve a catalog-backed high-level backtest path for optimization instead of forcing a low-level `BacktestEngine.reset()` rewrite.
- Add shared preflight validation and a `health` command to improve local repeatability.
- Retain a shared Nautilus log guard across repeated in-process backtest runs for optimization stability.

**Issues Resolved**

- Fixed CLI ergonomics around global `--config` option handling.
- Corrected Optuna workflow persistence and holdout export contracts.
- Stabilized repeated in-process backtest runs by retaining the Nautilus log guard.

**Issues Deferred**

- Intrabar or sub-minute execution simulation.
- MBP/order-book-driven absorption and richer microstructure logic.
- Multi-instrument, portfolio, or distributed optimization scope.

**Technical Debt**

- Phase directories remain in `.planning/phases/` for accessible raw execution history; archival cleanup can be done later if desired.
- No milestone audit artifact was created before archival, so v1.0 is archived based on completed execution and verification artifacts rather than a separate milestone-audit pass.
